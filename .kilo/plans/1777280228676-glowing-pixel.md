# Plan: Electron Codebase Security Audit Report

## Goal
Produce a detailed audit report documenting all security gaps, missing logic, and incorrect patterns in the deprecated Electron app (`/` root, not `web-prototype/`). The report is for reference only — no code changes to Electron.

## Context
- Electron app is **deprecated** per AGENTS.md; retained as logic reference only
- `web-prototype/` is the production target (offline-first PWA with IndexedDB)
- This audit documents what's missing/wrong so gaps aren't carried forward

## Output
A single file: `docs/electron-audit-report.md`

## Audit Findings to Document

### 1. Context Isolation & Node Integration (CRITICAL)
- `start.js:27-31`: `nodeIntegration: true`, `contextIsolation: false`
- No preload script exists
- Renderer (`pos.js`) directly requires: `electron`, `fs`, `path`, `crypto`, `@electron/remote`, `electron-store`
- Any XSS in the renderer gives full Node.js access (file system R/W, child_process, etc.)
- `@electron/remote/main` initialized and enabled for all `browser-window-created` events

### 2. IPC Channel Security (CRITICAL)
- No channel whitelisting — `ipcMain.on()` accepts any channel name
- `ipcRenderer.send("app-quit")`, `ipcRenderer.send("app-reload")`, `ipcRenderer.send("restart-app")` — no validation
- `ipcRenderer.on("click-element")` — main process can click ANY DOM element by ID, including triggering form submissions, modal opens, etc.
- No `ipcMain.handle()` pattern used — all synchronous `ipcMain.on()`
- No origin validation or payload sanitization on IPC messages

### 3. Authentication & Session Management (CRITICAL)
- Default credentials: `admin/admin` seeded via `GET /api/users/check/`
- `electron-store` stores auth state as plaintext JSON: `{ auth: true }` + full user object
- No session expiry, no token rotation
- Auth check in `pos.js` is client-side only — no server-side session validation per request
- `$.get(api + "users/check/")` has empty callback (line ~194) — appears to be dead code
- No password complexity requirements
- No account lockout after failed attempts
- Logout only sets user status to "Logged Out" in NeDB — doesn't invalidate the electron-store session

### 4. Content Security Policy (HIGH)
- CSP set via `<meta>` tag in renderer (weak — easily bypassed by any script injection)
- Should use `session.defaultSession.webRequest.onHeadersReceived()` in main process
- `'unsafe-eval'` and `'unsafe-inline'` in `script-src` completely negate hash-based protection
- SHA256 hashes computed but useless alongside `'unsafe-inline'`
- Stray double-semicolon (`; ;`) in policy string
- No `object-src`, no `media-src`, no `frame-src` restrictions
- `connect-src` allows `http://localhost:${PORT}` — acceptable for local API but no HTTPS enforcement

### 5. Auto-Update Implementation (HIGH)
- Uses `electron-updater` with generic provider — good
- `autoUpdater.autoDownload = false` — good, user consent required
- But: **no code signing verification** — `electron-updater` verifies signatures only if publisher is configured, and the config uses a generic HTTP endpoint without `cert` or `pub` key pinning
- `latest-yml.js` generates SHA-512 hashes but these are for integrity, not authenticity
- No forced TLS for update server URL (uses `https://` in config — OK, but no certificate pinning)
- `autoUpdater.quitAndInstall()` called without saving app state first

### 6. Renderer Process Crashes (HIGH)
- No `webContents.on('crashed')` handler
- No `webContents.on('unresponsive'/'responsive')` handlers
- No `gpu-process-crashed` handler
- No recovery mechanism — user must manually restart the app
- `process.on('uncaughtException')` in main process only logs to console — no user notification, no graceful shutdown

### 7. Backup/Restore Security (HIGH)
- Backup creates SHA-256 hash — good
- Restore verifies hash before extracting — good
- Directory traversal protection in restore (lines 238-243) — good
- BUT: no encryption on backup files — database contents are plaintext in ZIP
- No password protection on backup archives
- `restoreBackup()` calls `mainWindow.reload()` + `restartServer()` but doesn't validate the backup structure before applying it (if hash matches but contents are malformed, NeDB could corrupt)

### 8. Embedded Express Server (HIGH)
- CORS set to `*` — `res.header("Access-Control-Allow-Origin", "*")`
- No authentication on API routes — any process on the machine (or LAN, if port is exposed) can read/write all data
- Rate limiter: 100 requests per 15 minutes — very lenient
- No HTTPS — all data transmitted in plaintext over HTTP
- No request size limits on JSON body parser (default 100kb, but not explicitly configured)
- No input validation on many endpoints (e.g., `POST /api/transactions/new` doesn't validate most fields)
- `express-fileupload` with 2MB limit in inventory — but no virus scanning

### 9. Missing Application Lifecycle Handlers (MEDIUM)
- `before-quit` event not handled — no cleanup of: temp files, server connections, database connections, in-flight transactions
- No `will-quit` handler for cleanup
- No `window-all-closed` handler for macOS (exists but only quits on non-darwin — correct, but no window recreation guard)
- No single-instance lock (`app.requestSingleInstanceLock()`) — multiple instances could corrupt NeDB
- Server port not checked for availability before starting

### 10. Memory Leak Potential (MEDIUM)
- `server.js`: Express server and HTTP server never cleaned up on app quit
- `pos.js`: jQuery event handlers on `body` are never removed
- `product-filter.js`: jQuery `on()` handlers never cleaned up
- No `webContents.close()` cleanup
- `electron-store` instance never explicitly destroyed
- Socket.io imported in package.json but no visible usage — potential dead dependency

### 11. Missing Preload Script (MEDIUM)
- No preload script exists
- All Node.js APIs available in renderer: `fs`, `path`, `crypto`, `child_process` (implicit via `require`)
- No `contextBridge` API exposure — no API surface control
- Should use preload to expose only: `app-quit`, `app-reload`, `click-element`, `storage.get/set`

### 12. Command Line Arguments (MEDIUM)
- No command line argument sanitization
- No `app.commandLine.appendSwitch()` for security flags
- Missing: `--disable-blink-features=AutomationControlled`, `--no-sandbox` considerations
- Squirrel events handled but with empty catch blocks (line 22 of `setupEvents.js`)

### 13. Secure Storage (MEDIUM)
- `electron-store` uses unencrypted JSON file in `%APPDATA%`
- No `safeStorage` API usage for encrypting sensitive data
- Passwords in NeDB stored as bcrypt hashes — good
- But session tokens/auth state stored in plaintext
- Database files (`%APPDATA%/PharmaSpot/server/databases/*.db`) have no file system permissions set

### 14. Web Security Settings (MEDIUM)
- No `webPreferences.webSecurity: false` explicitly set — OK (defaults to true)
- No `webPreferences.allowRunningInsecureContent` — OK (defaults to false)
- No `session.clearCache()` / `session.clearStorageData()` on logout
- DevTools accessible in development mode — acceptable, but no guard against production exposure beyond `isPackaged` check

### 15. Multiple Window Scenarios (LOW)
- Only one `mainWindow` exists — no multi-window support
- No `BrowserWindow.getAllWindows()` usage
- `activate` handler creates new window if `mainWindow === null` — correct for macOS
- No protection against multiple instances modifying the same NeDB files

### 16. Network Connectivity (LOW)
- No online/offline detection in main process
- Renderer relies on jQuery AJAX error callbacks — no structured retry logic
- No request queuing for when connectivity is lost
- No visual indicator for server availability
- Server starts on fixed port 3210 — no fallback if port is occupied

### 17. Deep Links & Protocol Handlers (LOW)
- No custom protocol handler (`app.setAsDefaultProtocolClient`)
- No deep link support
- No file association handling

### 18. Logging (LOW)
- `console.log` / `console.error` only — no structured logging
- `electron-log` is a dev dependency but not used in runtime code
- No log rotation, no log levels, no log transport
- Server logs to console only

### 19. Environment Configuration (LOW)
- `app.config.js` only has UPDATE_SERVER and COPYRIGHT_YEAR
- No environment-specific config (dev/staging/prod)
- No feature flags
- `isPackaged` used as the only environment discriminator

### 20. Input Validation Gaps (MEDIUM)
- `validator.escape()` used on some fields in API routes but not consistently
- `DOMPurify.sanitize()` used for receipt rendering in pos.js — good
- No SQL injection protection needed (NeDB, not SQL) — OK
- But: No prototype pollution protection on JSON body parsing
- No request body schema validation

### 21. Native Menu Security (LOW)
- `handleClick(elementId)` accepts any string and calls `document.getElementById(elementId).click()` — could trigger unintended actions
- No allowlist of valid element IDs
- Menu items for Documentation and Send Feedback are empty stubs

### 22. Build & Packaging (LOW)
- `forge.config.js` has `asar: true` — good for basic source protection
- But `build.js` uses `asar: false` — inconsistency between build systems
- No code signing configured in Forge config
- No entitlements for macOS
- No Windows Authenticode signing
