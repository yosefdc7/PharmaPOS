## Objective

Execute **Path B**: fully remove the legacy Electron app from the repository, promote the repo to a web-prototype-first structure, and clean up documentation, tooling, and CI so Electron is no longer an active or implied runtime.

## Confirmed Findings

### Electron is still the active root app today

- Root `package.json` is still Electron-owned:
  - `main: start.js`
  - Electron dependencies in `dependencies` and `devDependencies`
  - Electron packaging/start scripts: `electron`, `electron-build`, `package-win`, `start`, `package`, `make`, `publish`
  - `credits: electronjs.org`
- Root CI still builds/releases Electron installers:
  - `.github/workflows/build.yml`
  - `.github/workflows/release.yml`

### Web app already has an independent delivery path

- `web-prototype/package.json` has its own scripts for `dev`, `build`, `start`, `test`, `typecheck`, `test:e2e`, and CI gates.
- `.github/workflows/web-prototype-delivery.yml` already runs web-prototype quality gates and staged delivery placeholders.

### Some root folders must not be blindly deleted

- `bridge/` is **not** Electron-only. `bridge/README.md` documents a LAN printer bridge used by the browser-based web app.
- `assets/` is referenced by `README.md` for the project logo (`assets/images/logo.svg`).
- `web-prototype` uses its own `/api/...` Next.js routes; that does **not** imply root `api/` must remain.

### Docs/rules still carry Electron/legacy language

- `AGENTS.md`
- `docs/PRD.md`
- `docs/TECH_STACK.md`
- `docs/DECISIONS.md`
- `docs/PRODUCTION_READINESS.md`
- `README.md`
- `.github/ISSUE_TEMPLATE/bug_report.md`

Validation:

- Search docs and project metadata for `electron|Electron|legacy|Legacy`.

### 2. Replace root `package.json` from Electron app package to workspace/meta package

Root `package.json` should stop being an Electron runtime package and instead serve as a repo entry point for the web prototype.

Planned changes:

- Remove:
  - `main: start.js`

  - `dev`
  - `build`

  - `typecheck`
  - optionally `test:e2e` / `ci:gates`

Reasoning:

- This preserves a coherent top-level developer experience after Electron removal.
- It prevents the repo root from becoming a broken or misleading package.

Validation:

- Root script inventory is internally coherent.

## Risks

### Medium risk

- Root package surgery can leave the repository with confusing or broken top-level scripts if not deliberately re-mapped.

### Medium risk

- Root `api/` deletion could remove logic still being informally referenced by docs or migration work, even if not runtime-critical.

### Low to medium risk

- `assets/` contains at least one README dependency, so deleting it wholesale would create broken docs.

### Low risk

- `bridge/` is clearly part of the web stack and should be preserved.

## Decisions Locked For Implementation

- Remove the Electron app and Electron-owned root packaging.
- Keep `web-prototype/` as the primary app.
- Keep `bridge/` unless implementation uncovers a contradiction.
- Do not delete `assets/` wholesale; audit and retain needed branding/docs assets.
- Replace root developer flow with wrappers into `web-prototype/`.
- Remove Electron CI/release workflows or rewrite them to web-only equivalents.

## One Small Open Choice During Execution

Choose one of these during implementation based on the cleaner result:

1. Keep a lightweight root `package.json` that delegates to `web-prototype/`.
2. Keep root `package.json` minimal and require developers to work directly inside `web-prototype/`.

Preferred option: **1** because it preserves a clean top-level UX and reduces repo friction.

## Follow-up Plan For Review Suggestions

The local review surfaced commit-hygiene issues rather than product/runtime regressions. These should be handled as a short follow-up cleanup pass after the Electron removal work.

### 10. Stop committing local Kilo task/runtime artifacts

Confirmed findings:

- Root `.gitignore` does **not** currently ignore `.kilocode/` task payloads/history.
- The review flagged many files under `.kilocode/tasks/...` and `.kilocode/task_history.json` as machine-local execution artifacts.

Planned changes:

- Add ignore rules for local Kilo task/runtime outputs at the repo root.
- Keep intentionally shared repo config under `.kilo/` untouched unless there is a specific reason to ignore a subpath.
- Remove the currently uncommitted `.kilocode/tasks/...` payload/history files from the commit set.
- Decide whether `.kilocode/task_history.json` is meant to be shared; default assumption should be **no** unless the team explicitly uses it as checked-in metadata.

Validation:

- `git status` no longer shows per-run `.kilocode/tasks/...` payload/history churn.
- No required project behavior depends on committed `.kilocode` runtime files.

### 11. Stop committing TypeScript incremental cache output

Confirmed findings:

- `web-prototype/.gitignore` ignores `.next/` but does **not** ignore `tsconfig.tsbuildinfo`.
- The review flagged `web-prototype/tsconfig.tsbuildinfo` as an ephemeral compiler artifact.

Planned changes:

- Add `*.tsbuildinfo` or a targeted `tsconfig.tsbuildinfo` ignore rule under `web-prototype/`.
- Remove the currently changed `web-prototype/tsconfig.tsbuildinfo` from the commit set.

Validation:

- Running typecheck no longer creates a tracked working-tree diff for `tsconfig.tsbuildinfo`.

### 12. Verify whether `web-prototype/next-env.d.ts` should remain tracked

Confirmed findings:

- `next-env.d.ts` now includes `import "./.next/dev/types/routes.d.ts";`
- `web-prototype/.gitignore` already excludes `.next/`, so the imported path is to generated dev-time output.

Planned checks:

- Confirm whether this line is an expected/generated change for the current Next.js version in this repo.
- Validate in a clean install/check flow whether `next-env.d.ts` is stable and safe to keep committed as generated.
- If it is transient machine-local output, revert it from the commit set and prevent recurrence if possible.
- If it is expected framework-managed output, leave it tracked but document that it is generated and should not be hand-edited.

Validation:

- Clean checkout / install / typecheck should not fail because of `next-env.d.ts`.
- The file should either stay stable across machines or be excluded from manual review noise by process/documentation.

### 13. Recommended execution order for the follow-up cleanup

1. Update ignore rules first (`.gitignore` and/or `web-prototype/.gitignore`).
2. Remove currently uncommitted Kilo task artifacts and `tsbuildinfo` from the working tree/index as appropriate.
3. Re-run `typecheck` to confirm `tsbuildinfo` no longer causes tracked churn.
4. Re-check `next-env.d.ts` after a clean framework-generated pass and decide whether to keep or drop the current change.
5. Run a final `git status` to confirm the commit contains only intentional source/docs/workflow changes.
