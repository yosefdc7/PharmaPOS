## Executive Summary

The recent changes (5 commits + 40 unstaged files) introduce a major refactoring of the sync engine, new security layers (encryption, sanitization, middleware CSP), pharmacy role-based access control, and a shift from per-entity sync to batch sync processing. **30+ TypeScript errors and 20 test failures** block production readiness. The core issue is a **schema version regression** (DB_VERSION 8ã6) combined with incomplete type/signature updates.

## Phase 2: Fix 20 Failing Tests

### 2.1 Schema Mismatch ã 15 test failures (db.auth.test.ts, db.indexes.test.ts, sync.integration.test.ts)

All fail with `DataError: Data provided to an operation does not meet requirements` at `seedIfNeeded ã putOne("users", ...)`.

**Root cause:** `putManyUsers` stores `StoredUser` objects but they lack `version` (removed in `saveLocalUserAccount` change). The `StoredUser` type requires `version`.

**Fix:** After making `version` optional on `User`/`StoredUser` (Phase 1.3), these tests will pass. Alternatively, the seed function can add `version: 1` to each seeded user.

`feature-flags.unit.test.ts` expects `DEFAULT_FEATURE_FLAGS` to be `{ sync: false, payments: false, refunds: false }` but the actual code has all `true`.

**Fix:** Update the tests to match the current defaults, OR revert the defaults to `false`. Based on the AGENTS.md (offline-first PWA), all flags should likely be `true` for the prototype. Update the tests.

### 2.3 Sync Worker Mock Missing `getByIndex` (3 failures)

`sync-worker.test.ts` mocks `./db` but doesn't include `getByIndex` which `runSync` now uses (changed from `getAll` to `getByIndex`).

## Phase 3: Security Hardening

### 3.1 CSP Headers (Priority: HIGH)

**File:** `web-prototype/src/middleware.ts`

- `'unsafe-eval'` and `'unsafe-inline'` in `script-src` severely weaken XSS protection
- Add `object-src 'none'`, `base-uri 'self'`, `form-action 'self'`
- Remove `ws:` / `wss:` from `connect-src` in production

### 3.2 API Route Authentication (Priority: HIGH)

Most API routes have **no authentication** on GET/PUT/DELETE:
- `categories/[id]`, `customers/[id]`, `products/[id]`, `transactions/[id]`, `settings`, `held-orders`
- Add `requireAuth()` to all entity CRUD routes
- For GET routes, consider read-only roles

### 3.3 Input Validation on API Routes (Priority: MEDIUM)

- `sync-queue/route.ts` ã Add column allowlist like `sync/process/route.ts` has
- All entity routes ã Validate body keys against known column maps
- `users/route.ts` ã Add `UNIQUE` constraint on username

### 3.4 Sanitization (Priority: LOW)

- `sanitize.ts` uses regex-based HTML stripping ã adequate for defense-in-depth but not primary XSS protection
- Ensure React's built-in escaping is the primary defense (it is by default)
- Consider adding DOMPurify for any `dangerouslySetInnerHTML` usage

### 3.5 Encryption Module Issues (Priority: MEDIUM)

**File:** `web-prototype/src/lib/encryption.ts`

- `cachedKey` singleton race condition ã add mutex/lock pattern
- Base64 encoding will fail on large payloads (>65KB) due to spread operator
- Key loss = permanent data loss ã add key export/backup mechanism or at minimum a user warning

## Phase 4: Data Integrity & Business Logic

### 4.1 Refund Flow Bug (Priority: HIGH)

**File:** `web-prototype/src/app/api/payments/refund/route.ts`

- `amount` parameter is destructured but never used (partial refunds unsupported)
- No stock restoration on refund ã products remain decremented
- UPDATE then SELECT is not atomic ã check `rowCount` before SELECT

### 4.2 Stock Decrement Silently Floors at 0

**File:** `web-prototype/src/lib/calculations.ts` ã `decrementStock()`

- `Math.max(product.quantity - sold, 0)` silently sets stock to 0 on oversell
- Should throw or warn when stock would go negative, especially for a pharmacy

### 4.3 SC/PWD Math Division by Zero

**File:** `web-prototype/src/lib/calculations.ts` lines 36, 70

- `vatRemoved / quantity` and `discountAmount / quantity` without zero check
- Add guard: `if (quantity === 0) return 0`

### 4.4 Z-Reading Compliance

- Z-Reading should be immutable once generated (override marks old one but doesn't prevent manipulation)
- Consider cryptographic signing or hash of Z-Reading data for audit integrity

### 4.5 Transaction Delete Route

**File:** `web-prototype/src/app/api/transactions/[id]/route.ts`

- BIR compliance requires transactions to be voided, not hard-deleted
- Remove or protect the DELETE handler

## Phase 7: Performance & Edge Cases

### 7.1 IndexedDB Indexes

- Verify V9 schema actually creates indexes on `syncQueue.status`, `transactions.createdAt`, `products.barcode`
- The `getByIndex` function requires these indexes to exist ã DB_VERSION 6 may not create them
- Either add index creation to V6 migration or bump version to 7/9

### 7.2 Audit Log Pagination

- `audit-trail.tsx` loads all audit entries via `getAll("auditLog")` ã will grow unbounded
- Add pagination or virtual scrolling

### 7.3 `makeLocalNumber` Uniqueness

- Uses `Date.now().slice(-5)` ã 5 digits with millisecond collisions possible
- Add counter or UUID suffix for uniqueness guarantee

### 7.4 Encryption Key Loss

- If IndexedDB `pharmapos-keys` is cleared, all encrypted sync queue items become unreadable
- Add user-facing warning and/or key backup mechanism
