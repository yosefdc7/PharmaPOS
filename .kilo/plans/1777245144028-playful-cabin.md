# Sync Remediation Plan — Critical Priority

## Problem Statement

The remote synchronization system (`sync-worker.ts`, API routes, IndexedDB queue) has **non-functional version tracking**, a **push-only architecture** with no pull sync, multiple **data integrity bugs**, and **zero test coverage** for all sync logic. Current status "partially resolved" — this plan brings it to full resolution.

---

## Phase 1: Fix Critical Foundation Bugs

### 1.1 Add `version` field to entity types and tracking

**Files:** `web-prototype/src/lib/types.ts`, `web-prototype/src/lib/db.ts`, `web-prototype/src/lib/server/schema.sql`

- Add `version: number` to `Product`, `Category`, `Customer`, `User`, `Settings`, `Transaction`, `HeldOrder` types
- Default `version: 1` on all new entity creation in `use-pos-store.ts` mutations
- Increment `version` on every mutation (`completeSale`, `saveProduct`, etc.)
- Add `version INTEGER DEFAULT 1` column to all entity tables in `schema.sql`
- Update `server/seed.ts` to include `version: 1` on all seeded rows

**Why:** Without a version field, `enqueueSync()` always assigns `entityVersion = 1` (via `NaN + 1` fallback), making version-based conflict detection completely non-functional. This is the root cause of the sync integrity failure.

### 1.2 Fix `enqueueSync` entityVersion calculation

**File:** `web-prototype/src/lib/db.ts` (~line 394-421)

- For "create" ops: `entityVersion = 1`
- For "update"/"delete" ops: `entityVersion = existing.version + 1` (now works because entities have `version`)
- Remove the `(existing as Record<string, number>).version` cast hack

### 1.3 Fix transaction syncStatus bulk update

**File:** `web-prototype/src/lib/sync-worker.ts` (~line 397-405)

Current code marks ALL pending transactions as synced after a sync run, including those whose sync items failed or conflicted. Fix:
- Collect IDs of successfully synced transaction items during `runSync()`
- Only mark those specific transactions as `"synced"` in IndexedDB
- Leave failed/conflicted items as `"pending"`

### 1.4 Fix delete-after-conflict-resolution bug

**File:** `web-prototype/src/lib/sync-worker.ts` (~line 250-272)

When a delete operation encounters a conflict and resolves as "remote-wins", the code still calls `pushToRemote(item)` to delete remotely — contradicting the resolution. Fix:
- If resolution is "remote-wins" on a delete, skip the push and mark item as "synced" (remote state preserved)
- Only push delete if resolution is "local-wins"

### 1.5 Fix manual strategy auto-merge on server-side

**File:** `web-prototype/src/app/api/sync/process/route.ts`

When `strategy === "manual"`, the server route falls through to `winner = "merged"` and auto-merges. Fix:
- When strategy is "manual", set item status to "conflict" and return it without auto-resolving
- The client's `resolveConflictManually()` path should be the only way to resolve manual conflicts

### 1.6 Add conflict column to contract test

**File:** `web-prototype/src/contracts/sync.contract.test.ts`

- Add `resolved_conflict` to the INSERT/SELECT round-trip test
- Validate it round-trips correctly as JSON

## Phase 2: Pull Sync (Server-to-Client)

### 2.1 Add `lastSyncTimestamp` tracking

**Files:** `web-prototype/src/lib/db.ts`, `web-prototype/src/lib/types.ts`

- Add a `syncMeta` IDB object store (or use existing `_meta` pattern) to persist `lastPullTimestamp` per entity type
- After a successful push sync, record `lastPullTimestamp = now`

### 2.2 Implement `pullFromRemote()`

**File:** `web-prototype/src/lib/sync-worker.ts`

New function `pullFromRemote()`:
- For each entity type, call `GET /api/{store}?since={lastPullTimestamp}`
- Server routes return entities with `updatedAt > since`
- Apply pulled entities to IndexedDB:
  - If entity doesn't exist locally → insert
  - If remote `version > local.version` → update (server wins by default for pull)
  - If versions conflict (local has pending sync) → enqueue as conflict
- Update `lastPullTimestamp` after successful pull

### 2.3 Integrate pull into `runSync()`

**File:** `web-prototype/src/lib/sync-worker.ts`

After push phase completes:
1. Call `pullFromRemote()`
2. Return combined report (push stats + pull stats)

### 2.4 Add `?since=` query param to entity API routes

**Files:** `web-prototype/src/app/api/products/route.ts`, `categories/route.ts`, `customers/route.ts`, `users/route.ts`, `transactions/route.ts`

- Accept optional `?since=<ISO8601>` query parameter
- Filter `SELECT` results where `updated_at > since`
- Required for the pull sync to fetch only newer entities

## Phase 3: Robustness & Resilience

### 3.1 Exponential backoff for retries

**File:** `web-prototype/src/lib/sync-worker.ts`

- Track `lastAttemptAt` on each SyncQueueItem
- Implement backoff: `min(2^retryCount * 5s, 5min)` — skip items that haven't waited long enough
- Items at max retries (5) stay as "failed" and don't block the queue

### 3.2 Network status detection & auto-sync trigger

**File:** `web-prototype/src/lib/sync-worker.ts`, `web-prototype/src/app/layout.tsx`

- Listen to `window.addEventListener('online', ...)` to trigger `runSync()` when connectivity returns
- Optionally add periodic sync interval (configurable, default 5 min) when `featureFlags.sync` is enabled
- Integrate with the existing Serwist service worker for Background Sync API where supported

### 3.3 Payload validation before push

**File:** `web-prototype/src/lib/sync-worker.ts`

- Add `validatePayload(entity, operation, payload)` that checks required fields per entity type
- Reject malformed payloads with descriptive error before making network calls
- Mark item as "failed" with validation error

### 3.4 Sync queue garbage collection

**File:** `web-prototype/src/lib/db.ts`

- Add `purgeSyncedItems(olderThanMs = 7 * 86400000)` — removes synced items older than 7 days
- Call after each successful `runSync()` completion

### 3.5 Fix bulk syncStatus marking — transaction granularity

Already addressed in 1.3. Ensure the fix is applied consistently to both IDB transactions and syncQueue items.

- [ ] All entity types have a `version` field that increments on every mutation
- [ ] `enqueueSync()` produces correct `entityVersion` values
- [ ] `detectConflict()` correctly identifies version mismatches for create/update/delete
- [ ] Only successfully synced transactions are marked as "synced"
- [ ] Delete operations respect conflict resolution (no contradictory push)
- [ ] Manual strategy leaves conflicts unresolved on server-side
- [ ] Pull sync fetches and applies remote changes since last sync
- [ ] Failed sync items use exponential backoff
- [ ] Sync triggers automatically when network comes back online
- [ ] Sync queue is pruned of old synced items
- [ ] All sync-worker functions have unit test coverage
- [ ] All sync API routes have test coverage
- [ ] End-to-end sync flow passes integration test
- [ ] Existing tests continue to pass (no regressions)
