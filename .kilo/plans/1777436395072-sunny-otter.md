# Blue-to-Teal Theme Rollout Plan

## Goal

Adopt the user's preferred blue-to-teal healthcare palette in the `web-prototype/` UI while keeping the app clean, clinical, and low-fatigue for daily POS use.

## Findings

- The shared theme tokens live in `web-prototype/src/app/globals.css`.
- The current accent system is centered on `--primary` and `--primary-dark`, which are consumed by many buttons, focus states, badges, and charts.
- Several purple values are still hard-coded outside the root tokens and would remain visually inconsistent unless updated.
- PWA/browser chrome colors are also hard-coded and need to be updated alongside UI tokens.

## Files Likely To Change

1. `web-prototype/src/app/globals.css`
2. `web-prototype/src/app/layout.tsx`
3. `web-prototype/public/manifest.json`
4. `web-prototype/src/app/~offline/page.tsx`
5. `web-prototype/src/components/override-modal.tsx`
6. `web-prototype/src/components/control-tower.tsx`

## Proposed Implementation

1. Update the root palette tokens in `globals.css`.
   - Replace the current purple `--primary` / `--primary-dark` with a blue-to-teal system.
   - Add one or two supporting tokens for translucent focus/hover states if needed, so shared UI stops relying on hard-coded RGBA values.

2. Keep most existing component styling intact by changing shared tokens first.
   - Preserve the current architecture where `.primary`, focus rings, selected states, and chart strokes derive from CSS variables.
   - This keeps the change small and prevents broad component churn.

3. Replace hard-coded purple accents that bypass tokens.
   - Update focus ring, sidebar active backgrounds, chart gradients, floating action shadows, and any other old purple RGBA values in `globals.css`.
   - Update direct hex usages in `layout.tsx`, `~offline/page.tsx`, and `override-modal.tsx`.

4. Align dashboard/chart accent usage.
   - Update the `control-tower.tsx` chart palette so the main accent no longer includes the old purple.
   - Keep other semantic colors like success/warning/danger unchanged unless they clash with the new palette.

5. Align PWA metadata.
   - Update `viewport.themeColor` in `layout.tsx`.
   - Update `theme_color` in `public/manifest.json`.
   - This ensures browser chrome and installed-app surfaces match the new theme.

6. Verify visual consistency.
   - Confirm primary buttons, focused inputs, sidebar active states, charts, and offline/restricted flows all match the new palette.
   - Check that contrast remains acceptable with the app's white/light surfaces.

## Recommended Palette Approach

- Base accent: trustworthy medical blue.
- Secondary accent / darker interaction color: deeper teal for hover, active, and gradient end states.
- Usage bias: keep large surfaces light and neutral; use the gradient sparingly so the interface stays clinical rather than decorative.

## Open Design Decision

Choose one implementation style before editing:

1. Conservative: use solid blue/teal tokens for buttons and controls, and reserve the gradient for brand surfaces only.
2. Stronger branding: use the blue-to-teal gradient directly on all `.primary` buttons and other primary CTAs.

## Recommended Path

Use the conservative option.

Reason:
- It fits the user's goal of a subtle, low-fatigue POS interface.
- It minimizes risk to text contrast and readability across the many existing `.primary` usages.
- It still allows gradient branding in select surfaces such as the brand mark, chart tint, or other accent areas.

## Verification

1. Run the web prototype and visually inspect shared controls affected by `--primary`.
2. Verify sidebar, buttons, inputs, focus rings, charts, offline page, and override modal.
3. Confirm installed/PWA theme color metadata reflects the new palette.

## Notes

- `docs/PRD.md` and `docs/TECH_STACK.md` do not appear to require updates because this is a visual palette refresh, not a scope or technology change.
- Seed/demo product color fields can remain unchanged unless brand consistency for demo data is also desired.
